# 502 root cause analysis — 2026-05-02

Investigated during the autonomous run while Noel was at the orchestra. Findings supplement the migration plan addendum.

## Where the 502 surfaces

**Not currently reproducing** (all probed net pages return 200 as of 2026-05-02 22:38 UTC). The 502 is INTERMITTENT, tied to the publisher's non-atomic file-write pattern.

## Architecture context

Production frontend (`https://www.blindhams.network/nets/...`) fetches net data client-side from `https://data.blindhams.network/next_nets.json` via JavaScript (CORS allowed by Apache's `Access-Control-Allow-Origin: *`).

Dev branch frontend uses a Netlify Function (`proxy-next-nets.js`) that proxies the same fetch server-side. **The function explicitly returns HTTP 502** when the upstream fetch fails:

```javascript
if (!resp.ok) {
  return { statusCode: 502, headers: jsonHeaders,
    body: JSON.stringify({ error: 'bad_gateway', status: resp.status, statusText: resp.statusText }) };
}
```

So a 502 visible on a "net page" means: the dev-branch frontend tried to fetch `next_nets.json`, Apache returned non-200, the function packaged that as a 502 to the user.

## What causes Apache to return non-200 on next_nets.json

The publisher cron runs `build_next_nets.py` every 5 minutes via `deploy_all.sh`. Per the WSL agent's findings, **the publisher writes via a non-atomic shell pattern** — write to a tempfile (`.next_nets.json.NNNNNN`), then rename to `next_nets.json`. If the rename step fails or the write doesn't complete cleanly, the publisher leaves zero-byte tempfiles AND `next_nets.json` is briefly missing or partial.

Evidence in `/var/www/data/`:
- 2,045+ leaked zero-byte tempfiles (`.bhn_nco_12w.json.NNNNNN` and `.next_nets.json.NNNNNN` patterns)
- The leaks accumulate every time the cron tick has a failure mode the wrapper doesn't handle

During the brief window when:
- The previous `next_nets.json` has been removed/renamed-away
- The new file isn't yet renamed into place

Apache returns 404. Production users see an empty next-nets section ("loading..." or empty list). Dev branch users see HTTP 502 from the proxy function.

## Why "broken for weeks"

The 502 is **probabilistic** — depends on whether a request hits Apache during the brief publish window. Most requests succeed; some don't. Over weeks, Noel saw 502s often enough to remember it as "broken." But it's never been consistently reproducible because the window is small and varies.

## The fix

**Q11's answer (locked 2026-05-02): rewrite the shell wrappers in Python with `os.replace()`.** That converts the file write to an atomic rename, eliminating the brief window where Apache sees a missing file. Tempfile leak becomes structurally impossible.

Specifically:
- Replace `build_bhn_12w.sh` and `build_next_nets.sh` with their contents folded into the corresponding Python scripts
- Each Python script writes to a tempfile in the same directory, then `os.replace(tmp, target)` for atomic POSIX rename
- No shell layer means no `trap`-needed cleanup, no leaked tempfiles
- Removes the 502 root cause AND the tempfile-leak operational issue in one change

## What the migration also fixes by side-effect

- `BHN_NETS_AUTO_PUSH=1` (Q1 answer) means helper writes flow through GitHub instead of staying local on Andre. The "WSL/rarbox divergence" risk dissolves.
- Cloudflare R2 + 60s TTL (architecture decision from this morning's chat) means user-side fetches go through CDN edge with cache buffer — even brief origin failures don't cascade to users (cache continues serving the prior version).

## Remaining concerns to flag

- The 2,045 existing leaked tempfiles in `/var/www/data/` still need cleanup. Migration plan covers this in decommission per Q10 ("Andre could care less, delete them when moving to rarbox"). Cleanup happens at decommission time, not now.
- Per the systemd helper config, the helper writes to `/home/ner/bhn/_data/nets.json` — different file than the publisher writes (`/var/www/data/next_nets.json`). The relationship is: publisher reads `/home/ner/bhn/_data/nets.json` to generate `/var/www/data/next_nets.json`. So the helper's writes feed into the publisher's data flow, but the user-visible file (`next_nets.json`) is the publisher's output.
