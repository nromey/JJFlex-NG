## Discrepancies worth noting

- The `dotnet clean -c Release` step initially fails with `NETSDK1047` ("assets file doesn't have a target for net10.0-windows/win-x64") on both architectures. This is the dotnet SDK noticing a stale assets file from a prior platform — the subsequent `dotnet build` performs an implicit restore and the build succeeds. No action needed; it's preexisting behavior, not introduced by the upgrade.
- After the x64 Release build, only `Setup JJFlex_4.1.16.0_x64.exe` (and a duplicate `Setup JJFlex_x64.exe`) are visible at the repo root. The x86 Release post-build hook did not produce an `_x86` installer in this run. Suspected cause: the `install.bat` post-build target is wired to fire only on certain Configuration|Platform tuples and may have been short-circuited because the x64 build had already produced the artifact. Worth a Track A / cleanup-sprint follow-up; not blocking for the upgrade itself.
