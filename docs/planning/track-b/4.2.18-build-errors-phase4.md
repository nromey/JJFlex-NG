## Why Phase 5 may be a no-op

The Phase 1 inventory anticipated call-site work for renames and signature changes:

- `radioShouldUseExtTuner` → `ShouldUseExternalTuner`
- `UDPVitaPacket` → `UdpVitaPacket`
- `IsTxAmpModel` removed
- `SendUpdateFile` → async
- `WaveformsInstalledList` → `ObservableCollection<Waveform>`
- `LocalIP` → `LocalIp`
- `AmplifierFault` → `HaapiFault`
- `EqualizerStatus` record gains `rfpower`

A repo-wide grep for these symbols (excluding `FlexLib_API/`, `flexlib4218/`, `bin/`, `obj/`) returned zero hits before the build, and the build now confirms it: JJFlex never consumed any of those surfaces directly. They are vendor-internal or vendor-public-but-JJFlex-doesn't-use.

The two surfaces JJFlex *does* heavily consume — `Slice`, `Meter`, `Tuner`, `FeatureLicense`, `Radio` (the unchanged parts), `WanServer`, plus the discovery/connect machinery — are either unchanged in public surface (Slice +3 LOC, Meter +1 LOC, Tuner +6 LOC), or the public-surface additions are purely additive (license flags, filter presets, NAVTEX, display markers).

## Implications for Phase 5

- **No call-site changes required for the v4.0.1 → v4.2.18 upgrade.**
- Phase 5 commit will be empty (or absent).
- The vendor-side TLS / Discovery patches in Phase 3 are sufficient to keep behavior consistent with what we shipped under v4.0.1.
- New surfaces (filter presets, NAVTEX, display markers, DAX RX list, WFP state, IARU/Turf region, etc.) can be opted into during future feature sprints without blocking this upgrade.

## Things still needing real-radio verification (Track A)

A clean build is necessary but not sufficient — these need a live radio:

- TLS negotiation logs the chosen protocol (look for `SslClientTls12 negotiated protocol: Tls13` in Debug output).
- TLS-on-LAN behavior, since Phase 3's Radio.cs patch now selects TLS whenever `PublicTlsPort > 0` (was `IsWan` only). On older firmware that doesn't expose `PublicTlsPort`, this is a no-op; on current firmware it shifts LAN connections to the TLS path. Verify a normal LAN radio connects.
- Discovery race fix — re-trigger `apiInit(force=true)` (e.g., a network adapter change while connected) and confirm no NRE.
- Audio paths — the Vita pipeline got new schedulers (HighPriorityTaskScheduler, MmcssPipelineScheduler) and a VitaPacketBase refactor. RX, TX, DAX paths need exercise.
- AS-retry pathway — `project_as_retry_pathway_regression.md` notes this got jankier in Sprint 26+27. With v4.2.18, see whether the secondary retry behaves better, worse, or the same.
- HAAPI — if Noel ever exercises an external amplifier, the rewritten event surface (HaapiFault/HaapiWarning, per-meter data-ready events) becomes load-bearing. JJFlex doesn't subscribe to any of this today, so behavior is unchanged but worth a single connect with the amp on, just to confirm no startup error.
