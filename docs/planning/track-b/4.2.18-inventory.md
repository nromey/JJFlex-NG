# FlexLib 4.0.1 → 4.2.18 inventory

Map of vendor surface changes between `FlexLib_API/` (current v4.0.1) and `flexlib4218/` (vendor v4.2.18) to inform Phase 2 (replace), Phase 3 (patches), and Phase 5 (call-site fixes).

## File-level summary

- v4.0.1: 133 files
- v4.2.18: 139 files
- Identical: 22 files
- Modified: 109 files
- New in v4.2.18: 8 vendor files
- "Removed" in v4.2.18: 2 files, neither a true vendor remove
  - `Directory.Build.props` — vendor moved it to repo root of v4.2.18 source (`flexlib4218/Directory.Build.props` is currently absent in our staging dir, so v4.2.18 may ship without it; we will keep the existing `FlexLib_API/Directory.Build.props` unless it conflicts)
  - `FlexLib/SslClientTls12.cs` — our custom TLS wrapper, reapplied in Phase 3

So the upgrade is additive on the vendor side — zero true removals — but the public-API surface of multiple core files has shifted.

## New files in v4.2.18

FlexLib core:
- `FlexLib/DisplayMarker.cs` — panadapter/waterfall marker objects
- `FlexLib/Filter.cs` — Filter type extracted as standalone class (filter-preset API on Radio depends on it)
- `FlexLib/NAVTEX.cs` — NAVTEX (Navigational Telex) feature support
- `FlexLib/SsdrErrors.cs` — error type unification
- `FlexLib/Waveform.cs` — waveform plug-in metadata (replaces flat string list of installed waveforms)

Vita pipeline:
- `Vita/HighPriorityTaskScheduler.cs` — task scheduler used by audio paths
- `Vita/MmcssPipelineScheduler.cs` — Multimedia Class Scheduler Service integration for low-latency audio
- `Vita/VitaPacketBase.cs` — base class refactor for Vita packet types

JJFlex consumes none of these directly today. They are vendor additions; opt-in if/when we expose new features. NAVTEX/Waveform/DisplayMarker may eventually feed Customize Home and waterfall work but are out of scope for this upgrade.

## Highest-risk modified files (call-site impact)

These are the files where Phase 5 work concentrates. Each is annotated with the public API drift discovered by `diff` plus a JJFlex-impact note.

### Radio.cs (+679 LOC)

The central type — biggest API drift surface.

Renames JJFlex consumes:
- `radioShouldUseExtTuner` → `ShouldUseExternalTuner` (property rename)
- `UDPVitaPacket` → `UdpVitaPacket` (nested class rename, plus ctor name)
- `UDPCallbackQueue` field type changes accordingly

Removed:
- `IsTxAmpModel` — replaced by `IsSystemModel` (different semantic — verify before rewiring)

Signature changes:
- `UninstallWaveform(string)` → `UninstallWaveform(string, bool is_docker = false)` (backward-compatible — optional param)
- `SendUpdateFile(string)` → `async Task SendUpdateFile(string)` (now async — call sites that fire-and-forget need `_ = SendUpdateFile(...)` or `.GetAwaiter().GetResult()`; await if reachable)
- `WaveformsInstalledList` type changed from `ObservableCollection<string>` to `ObservableCollection<Waveform>`
- `FindWaterfallByParentStreamID` accessibility: `internal` → `public` (no impact)

New API surface:
- Filter presets — `GetFilterPreset`, `GetFilterPresetGroup`, `UpdateFilterPreset`, `SaveFilterPreset`, `ResetFilterPreset`, `FilterPresets`, `NUM_FILT_PRESETS`
- Display markers — `DisplayMarkers`, `DisplayMarkersLockObj`, `AddDisplayMarker`, `RemoveDisplayMarker`, `DisplayMarkerAdded`, `DisplayMarkerRemoved` events
- DAX RX list — `DAXRXAudioStreamList`
- `NAVTEX NAVTEX => _navtex`
- `IARURegion`, `TurfRegion`
- WFP power state — `IsWfpPowered`, `IsWfpReady`, `WfpIPAddress`
- `RemoveAllPanadapters()`, `RemoveAllWaterfalls()` (internal — no JJFlex impact)
- `IsSystemModel` (likely partial replacement for `IsTxAmpModel`)

JJFlex consumers to verify in Phase 5: `Radios/FlexBase.cs`, `Radios/AllRadios.cs`, anywhere using `Radio` directly.

### HAAPI.cs (+345 LOC)

Substantially redesigned.

Removed:
- `AmpIsSelected` — gone, no direct replacement (HAAPI now has multi-meter API instead)
- `AmplifierFault(string reason)` event delegate — replaced by `HaapiFault(string noun, string reason)` with extra noun parameter
- `AmplifierFault` event — renamed to `HaapiFault`

Added:
- `HaapiWarning`, `HaapiWarningCleared` events (new fault/warning channel)
- Per-meter data-ready events: `HaapiFwdPwrDataReady`, `HaapiVswrDataReady`, `HaapiHVDataReady`, `HaapiCurrentDataReady`, `HaapiTempPsuDataReady`, `HaapiTempPa0DataReady`, `HaapiTempPa1DataReady`, `HaapiTempDrvADataReady`, `HaapiTempDrvBDataReady`, `HaapiTempCombDataReady`, `HaapiTempHpfDataReady`
- `SendHaapiCommand(string)`, `HaapiChangeMode(AmplifierMode)`
- `FindMeterByIndex`, `FindMeterByName`, `AddMeter`, `RemoveMeter`

JJFlex impact: low. JJFlex has no current grep hits for `HAAPI`, `AmpIsSelected`, or `AmplifierFault`. PowerGenius / external amplifier UI is not currently exposed. If anything hooks `AmplifierFault`, it must move to `HaapiFault` and accept the extra `noun` parameter.

### APD.cs (+291 LOC)

Auto Power-down / equalizer expanded.

Public-record signature change:
- `EqualizerStatus(bool enable, string ant, double freq)` → `EqualizerStatus(bool enable, string ant, double freq, int rfpower)`. Any pattern-match or positional construction in JJFlex breaks.

Added:
- `APDSamplerPorts` enum
- `EqualizerReset()` method
- Per-antenna sampler port lists and selectors: `AvailableSamplerPortListANT1/2/XVTA/XVTB`, `SelectedSamplerPortANT1/2/XVTA/XVTB`
- `EqualizerActiveHeartbeat`, `EqualizerCalibratingHeartbeat` events

JJFlex impact: low to moderate. Verify whether anything uses `EqualizerStatus` positionally — if so, add the `rfpower` field handling.

### Slice.cs (+3 LOC)

Effectively unchanged in public surface — small delta. Slice is JJFlex's most consumed type, so this is low risk. Still worth a content scan for non-LOC behavioral tweaks.

### Meter.cs (+1 LOC)

Single addition: `public const string SOURCE_HA_API = "HAAPI"`. No call-site impact.

### FeatureLicense.cs (+76 LOC)

License flag taxonomy expanded.

Added flags:
- `LicenseFeatNAVTEX`
- `LicenseFeatRapidmLogging`
- `LicenseFeatPanadapterVisuals`
- `LicenseFeatFilterPresetConf`

JJFlex impact: none unless we add UI for any of them. Existing flag accessors (`LicenseFeatDivEsc`, etc.) are unchanged.

### Discovery.cs (+15 LOC)

Vendor v4.2.18 has independently absorbed *some* of our race-fix work:
- `Stop()` now does `udp?.Close()` (good — improvement)
- `Receive()` now catches `OperationCanceledException`, `ObjectDisposedException`, and `SocketException` filtered on `OperationAborted`

Vendor still has the static-`udp`-null race we patched:
- `Receive()` uses static `udp` directly (no local capture)
- No null-guard at task entry
- Task unconditionally nulls `udp` at end (line 125 in v4.2.18)

So our patch is still needed. Phase 3 reapplies the local-capture, null-guard, and only-null-if-still-own elements. The vendor-side catches are now redundant with ours but harmless.

### TlsCommandCommunication.cs (+82 LOC)

Substantial vendor refactor:
- Now file-scoped namespace + `#nullable enable`
- Property rename `LocalIP` → `LocalIp`
- Field type `_tlsToRadio: SslClient` (still) — our patch still swaps to `SslClientTls12`
- Connect flow changed — vendor's `SslClient` no longer connects in ctor; uses `Task<bool> Connect()` instead, awaited via `.GetAwaiter().GetResult()`
- Disconnected event handler signature changed: `(object?, bool)` instead of `(object, EventArgs)`
- Event-handler delegate cased differently: `TcpDataReceivedReadyEventHandler` (was `TCPDataReceivedReadyEventHandler`)

### SslClient.cs (vendor — +28 LOC)

Wholesale rewrite:
- Async `Task<bool> Connect()`
- camelCase ctor params (`startPingThread`, `validateCert`)
- `Disconnected` event signature `EventHandler<bool>` (was `EventHandler`)
- Uses `SslClientAuthenticationOptions { TargetHost = _hostname }` with default `EnabledSslProtocols = SslProtocols.None` — no explicit TLS floor. **This is exactly why our wrapper still matters** — without enforcement we get whatever the OS picks.

Phase 3 implication: our `SslClientTls12.cs` wrapper needs a refresh to mirror the new `SslClient` API surface (async Connect, camelCase, `EventHandler<bool>` Disconnected) so the swap in `TlsCommandCommunication.cs` is a one-token rename.

## Phase 5 likely batches

Anticipated commit batches (final grouping decided once Phase 4 errors come in):

1. **Renames** — `radioShouldUseExtTuner` → `ShouldUseExternalTuner`, `UDPVitaPacket`/`UDPCallbackQueue` casing, `LocalIP` → `LocalIp`. Pure mechanical.

4. **`WaveformsInstalledList` type change** — call sites that read strings need to read `Waveform.Name` (or equivalent) instead.
5. **`HAAPI.AmplifierFault` rename** (only if anything subscribes — none today).
6. **Mock surface updates** — `Radios.Tests/MockWanServer.cs` if it conforms to a changed interface.

Anything not anticipated falls out of the Phase 4 error list and gets its own batch.
