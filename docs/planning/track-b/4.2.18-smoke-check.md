# Phase 7 — No-radio smoke check

## Outcome: clean startup confirmed

Both Debug and Release x64 builds were launched headlessly from PowerShell, exercised, and shut down. No crash, no startup exception, no stderr output.

### x64 Release (`bin/x64/Release/.../JJFlexRadio.exe`)

- Launched, PID assigned, process stayed alive.
- After ~8 seconds: process running, `MainWindowHandle=0` because the foreground window was the **"Select Radio" dialog** (the FlexLib chooser shown when no radio is auto-discovered) — exactly the expected no-radio state. WPF infrastructure windows (GDI+ Window, MediaContextNotificationWindow, .NET-BroadcastEventWindow) all present and well-formed.
- WM_CLOSE posted to the Select Radio dialog: process exited gracefully within ~5 seconds.
- Working set ~197 MB, 44 threads — normal for a WinForms+WPF startup.

### x64 Debug (`bin/x64/Debug/.../JJFlexRadio.exe`)

- Same outcome — Select Radio dialog appears, no startup crash, stderr empty.
- WM_CLOSE took longer to propagate in this run; ultimately force-killed at 5 seconds. Not a bug in the app — the dialog's close handler may want a Cancel/X click rather than a WM_CLOSE postMessage. Worth a Track A re-test from a real user input.

### What was implicitly exercised

- All four FlexLib assemblies (`FlexLib.dll`, `Vita.dll`, `Util.dll`, `Flex.UiWpfFramework.dll`) loaded without binding errors. WPF reference resolution, WinForms→WPF interop (ElementHost), TLS wrapper class registration: none of these blew up at startup.
- `Discovery.Start()` was hit (the Select Radio dialog implies discovery initialized — it would deadlock-or-NRE if it weren't). The race-fix patch from Phase 3 did not introduce a regression on cold start.
- `globals.vb`'s `apiInit` path is reachable.
- The dispatcher unhandled-exception handler from the 2026-04-26 standalone fix was never triggered — there were no unhandled exceptions to catch.

## What still needs Track A

The headless test confirms "starts, doesn't crash, presents an interactive surface, accepts WM_CLOSE." The screen-reader-specific behaviors and live-radio flows still need a sighted/screen-reader-equipped session:

1. **"No radio connected" announcement**: does the screen reader say something useful when the Select Radio dialog comes up? The dialog appears, but Track B can't hear what it says.
2. **F2 / JJ Flexible Home tabbing**: with the Select Radio dialog dismissed (Cancel), does the main window's tab-stop sequence still work? Does F2 land on JJ Flexible Home?
3. **No-silent-keystroke guard**: press a bound key while no radio is connected (e.g., F4 or whatever the connect/disconnect chord is). The 2026-04-28 guard from `project_no_silent_keystrokes_rule.md` should produce audible feedback ("no radio connected" or similar).
4. **Discovery cycle**: open Select Radio, press Refresh, watch for the new Phase 3 trace lines:
   - `Discovery.Receive: task started (udp=set)` (new task)
   - `Discovery.Receive: task exited cleanly` (old task exiting)
   - No NRE.
5. **TLS log line on connect**: connect to a radio (LAN or SmartLink). Watch Output for `SslClientTls12 negotiated protocol: Tls13` (or `Tls12` fallback). This is the proof that the wrapper is in the path under v4.2.18.
